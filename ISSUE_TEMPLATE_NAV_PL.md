**Feature Request: Add support for UBX-NAV-PL (Protection Level) message**

## Issue Body

### Summary

Add support for parsing the UBX-NAV-PL (0x01 0x62) message, which provides Protection Level information for GNSS positioning. This message is essential for safety-critical applications requiring integrity information about position accuracy bounds.

### Motivation

The UBX-NAV-PL message provides protection level information that bounds the true position error with a specified confidence level. This is crucial for:

- **Autonomous vehicles** requiring integrity monitoring
- **Aviation applications** needing horizontal/vertical protection levels
- **Safety-critical systems** where position uncertainty quantification is mandatory
- **Compliance with standards** like DO-229 (aviation) or automotive safety standards

### Message Specification

**Fixed Payload Length:** 56 bytes

### Protocol Version Support

NAV-PL availability varies by device family (not just protocol version):

| Device | Firmware | Protocol Version |
|--------|----------|------------------|
| ZED-F9P | HPG 1.30+ | 27 |
| ZED-F9R | HPS 1.30+ | 33 |
| M10 | SPG 5.10+ | 34 |
| NEO-F10N | SPG 6.00+ | 40 |

**Note:** X20/F20 devices do **not** appear to support NAV-PL currently.

#### Recommended Approach

Since `ubx_proto27` is already supported and ZED-F9P (a widely-used device) has NAV-PL at protocol 27, the pragmatic approach is:

1. **Add NAV-PL to `ubx_proto27`** - This enables support for ZED-F9P users immediately
2. **Also add to `ubx_proto31`** - For consistency with higher protocol versions
3. **Document device-specific availability** - Users must verify their device supports this message

The protocol version feature flags in ublox-rs represent "messages available at or after this protocol version" rather than strict per-device compatibility. Users are expected to know their device's capabilities.

Alternatively, a new feature flag like `nav_pl` could be introduced for explicit opt-in, but this adds complexity and diverges from the existing pattern.

### Proposed Implementation

1. Create new file `ublox/src/ubx_packets/packets/nav_pl.rs`
2. Define `NavPl` struct using `#[ubx_packet_recv]` macro
3. Add appropriate validity flag enums/bitflags
4. Register in relevant `packetref_proto*.rs` files
5. Export from `packets.rs`
6. Add unit tests
7. Update documentation/CHANGELOG
